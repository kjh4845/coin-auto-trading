# BTCUSDT 선물 AI 자동매매 봇 청사진

## 1. 목표

바이낸스 `BTCUSDT` 무기한 선물 전용 자동매매 봇을 만든다.
조건은 다음과 같다.

- 거래 대상은 `BTCUSDT` 하나만 사용
- 초단타 스캘핑이 아니라 `수 분 ~ 수 시간` 보유하는 단타
- 관련 있는 비트코인 시장 데이터는 가능한 한 최대한 많이 수집
- 로컬 `Gemma4`는 보조 판단기로 사용
- 손절은 `포지션에 사용한 초기 증거금의 -5%` 기준
- 익절은 `ATR 기반 트레일링 스탑` 중심으로 운영
- 최대 레버리지는 `10배`까지 지원
- 남는 M2 맥북 에어에서 로컬 추론으로 안정적으로 동작

1차 목표는 수익 극대화가 아니다.
우선순위는 다음과 같다.

- 정확한 데이터셋 구축
- 재현 가능한 백테스트
- 리스크 엔진 검증
- 실거래 전에 테스트넷 실행 검증

## 2. 핵심 원칙

### 2.1 AI가 계좌를 직접 지배하면 안 된다

Gemma4가 할 수 있는 일:

- 시장 국면 분류
- 셋업 품질 점수화
- 낮은 품질의 진입 거부
- 익절 구간에서 스탑을 더 타이트하게 가져갈지 제안
- 매매 이유 요약

Gemma4가 하면 안 되는 일:

- 레버리지 제한 우회
- 손절 규칙 무시
- 리스크 매니저 없이 직접 주문
- 손절 폭 확대
- 킬 스위치 무시

최종 권한 흐름은 아래와 같다.

`Signal Engine -> Risk Manager -> Execution Engine`

### 2.2 데이터는 넓게 모으고, 학습은 구조화해서 한다

비트코인 관련 데이터는 최대한 많이 모아도 된다.
예를 들면:

- 선물 계약가격 OHLCV
- 마크프라이스
- 펀딩비
- 미결제약정
- 테이커 매수/매도 비율
- 필요하면 현물 BTC 데이터

하지만 Gemma4에 원시 차트 데이터를 무한정 던져서 학습시키는 방식은 좋지 않다.
실제로는 이렇게 가야 한다.

- 시장 데이터는 폭넓게 저장
- 그 데이터를 학습 샘플 단위로 가공
- 일정 길이의 시장 윈도우를 잘라서 라벨링
- 추론 시에는 작은 피처 스냅샷만 입력

### 2.3 시그널 가격과 리스크 가격은 분리한다

하나의 가격 소스를 모든 목적에 쓰면 안 된다.

권장 규칙:

- 시그널 생성은 계약가격 기반의 실제 실행 가능한 데이터 사용
- ATR, 구조, 모멘텀, 진입 타이밍은 계약가격 캔들 사용
- 하드 손절 트리거, 미실현 손익, 청산 위험 체크는 마크프라이스 사용
- 사후 리뷰에는 계약가격과 마크프라이스 둘 다 저장

짧게 정리하면:

`signals on contract price, risk on mark price`

### 2.4 버전 1에서는 전략을 하나만 쓴다

버전 1 전략은 하나로 고정한다.

`추세 추종형 눌림목 진입 + 변동성/시장국면 필터`

처음부터 아래를 섞지 않는다.

- 돌파
- 역추세
- 평균회귀
- AI 완전자율 진입

### 2.5 주문 경로는 항상 설명 가능해야 한다

모든 실거래 주문은 아래 질문에 답할 수 있어야 한다.

- 왜 진입했는가
- 왜 그 사이즈를 썼는가
- 손절이 어디에 있었는가
- 왜 손절이 이동했는가
- 왜 청산되었는가

### 2.6 라이브 코어 룰은 불변이어야 한다

리뷰와 재학습 루프는 변경안을 제안할 수 있지만, 원래 정한 라이브 정책을 조용히 약화시키거나 삭제하면 안 된다.

버전 1의 불변 코어 룰:

- 거래 종목은 `BTCUSDT` 하나만 사용
- 마진은 `isolated`만 사용
- 승인된 레버리지 상한 초과 금지
- 동시 포지션은 1개만 유지
- 진입 시점에 고정한 초기 증거금 기준 `-5%` 하드 손절 유지
- 하드 손절 트리거는 마크프라이스 기준 유지
- 항상 보호 스탑이 존재해야 함
- 마틴게일과 무제한 물타기 금지
- 실거래 중 자기 룰을 스스로 바꾸는 기능 금지

리뷰 시스템이 할 수 있는 것:

- 더 타이트한 필터 제안
- 더 안전한 파라미터 범위 제안
- 추가 진단 로직 제안
- 오프라인 검증용 후보 전략 버전 제안

리뷰 시스템이 하면 안 되는 것:

- 하드 손절 삭제
- 레버리지 한도 자동 확대
- 보호 스탑 의무 제거
- 새 전략 계열을 라이브에 직접 활성화
- 별도 승인 없는 분할매수 활성화

## 3. 매매 스타일 정의

이 봇은 다음 스타일을 목표로 한다.

- 보유 시간은 수 분에서 수 시간
- 매 틱 반응이 아니라 캔들 마감 기준 판단
- 실행 속도는 중요하지만 HFT 인프라는 불필요

권장 타임프레임:

- 실행 타임프레임: `3m`
- 확인 타임프레임: `15m`
- 상위 추세 필터: `1h`

추후 확장:

- 필요하면 `5m` 확인 레이어 추가

## 4. 시장 및 계좌 제약

### 4.1 거래 범위

- 거래소: Binance USD-M Futures
- 심볼: `BTCUSDT`
- 계약: 무기한 선물
- 버전 1은 `one-way mode`
- 버전 1은 동시 포지션 1개만 허용
- 마진 모드는 `isolated`

### 4.2 레버리지

- 시스템이 지원하는 최대 레버리지: `10x`
- 실거래 시작 권장 레버리지: `2x ~ 3x`
- 충분한 테스트넷 및 소액 실거래 검증 후에만 상향

중요한 점:

- `증거금 기준 -5% 손절`은 레버리지가 올라갈수록 매우 타이트해진다
- `10배`에서는 대략 `0.5%` 정도의 불리한 가격 움직임만으로 손절될 수 있다

즉, 이 규칙은 반드시 백테스트로 버틸 수 있는지 확인해야 한다.

### 4.3 포지션 사이징

사이징은 레버리지 한도보다 계좌 리스크 기준으로 정해야 한다.

권장 통제값:

- 1회 거래 리스크: 계좌의 `0.5% ~ 0.75%`
- 일일 실현손실 한도: `2%`
- 주간 드로우다운 중지선: `5%`
- 동시 포지션: 1개
- 거래 종목: 1개

사이즈 산정 요소:

- 진입가
- 사용할 초기 증거금
- 레버리지 상한
- 거래소 최소 수량/최소 주문금액
- 손절 규칙

## 5. 손절 정책

### 5.1 하드 스탑 정의

이번 기준에서는 손절을 이렇게 정의한다.

`마크프라이스 기준 미실현 손익이 진입 시점에 고정된 초기 증거금의 -5% 이하가 되면 강제 청산`

여기서 `-5%`는 최종 실현손실 상한이 아니라 손절 트리거 기준이다.
슬리피지, 수수료, 갭, 지연 체결 때문에 실제 실현손실은 더 커질 수 있다.

이 값은 진입 체결이 확정되는 순간 고정해서 저장한다.

공식은 다음과 같다.

- `filled_entry_notional = average_fill_price * filled_quantity`
- `entry_initial_margin_fixed = filled_entry_notional / leverage_at_entry`
- `hard_stop_loss_usdt = entry_initial_margin_fixed * 0.05`

트리거:

- 롱/숏 모두 `mark_price_unrealized_pnl <= -hard_stop_loss_usdt` 이면 즉시 청산

### 5.2 이 기준의 의미

이 손절은 다음 기준이 아니다.

- 가격이 몇 퍼센트 역행했는가
- 계좌 전체가 몇 퍼센트 빠졌는가
- ATR 기준으로 얼마 움직였는가

정확히는 `진입 시점에 고정된 초기 증거금 계약` 기준이다.
그래서 레버리지가 높을수록 손절은 더 촘촘해진다.

바이낸스 포지션 응답의 현재 `initialMargin` 또는 `positionInitialMargin` 값을 이 규칙의 기준으로 쓰면 안 된다.
공식 문서상 이 값들은 현재 마크프라이스 기준 필요 증거금이기 때문에, 네가 원하는 "진입 시점 고정 증거금"과 다르다.

### 5.3 실전 구현 방식

손절은 2중 구조로 간다.

1. `거래소 측 보호 스탑`
- 진입 체결 직후 `STOP_MARKET reduce-only` 손절 주문 배치
- 고정된 증거금 손절 규칙을 가격 레벨로 환산해서 등록
- 조건 트리거는 `workingType=MARK_PRICE` 사용

2. `로컬 fail-safe 체크`
- 실행 캔들 마감 시점과 주요 계정 이벤트 때마다 재검산
- 마크프라이스 기준 미실현 손익을 다시 계산
- 조건 충족 시 손절 주문 상태와 무관하게 강제 청산
- 이미 임계값을 넘었는데 거래소 손절이 체결되지 않으면 reduce-only `MARKET`로 즉시 강제 청산

### 5.4 추가 손절 안전장치

- 보호 스탑 주문이 실패하면 즉시 포지션 정리
- 보호 스탑이 있어도 로컬에서 마크프라이스 손절 위반이 확인되면 시장가로 정리
- 로컬 상태와 거래소 상태가 다르면 신규 진입 중단
- 손절 슬리피지가 반복적으로 커지면 거래 중단 또는 사이즈 축소

## 6. 익절 및 청산 정책

### 6.1 기본 철학

익절은 고정 목표가보다 `ATR 트레일링 스탑` 중심으로 가져간다.

즉:

- 진입 시 하드 손절은 명확하게 둔다
- 수익이 나는 동안은 따라간다
- 추세가 꺾이면 ATR 기반 스탑으로 청산한다

권장 처리 방식:

- 거래소 보호 스탑은 마크프라이스 하드 손절 계약 전용으로 유지
- ATR 트레일은 전략 청산 로직이지, 기본 안전 스탑 자체는 아님
- ATR 또는 시간 스탑이 청산을 요구하면 로컬 엔진이 reduce-only 청산 주문을 실행

### 6.2 ATR 트레일링 스탑 정의

버전 1 권장값:

- ATR 계산: 실행 타임프레임의 계약가격 캔들 기준 `ATR(14)`
- 기본 배수: `2.5`

롱 포지션:

- `trail_stop = 진입 후 최고가 - ATR(14) * 2.5`

숏 포지션:

- `trail_stop = 진입 후 최저가 + ATR(14) * 2.5`

규칙:

- 캔들 마감 때만 갱신
- 스탑은 타이트해질 수는 있어도 느슨해지면 안 됨
- ATR 로직은 시그널/청산 보조 로직이며, 마크프라이스 하드 손절을 대체하지 않음
- 기본 구현은 캔들 마감 시점 reduce-only `MARKET` 청산
- 추후 최적화로 짧은 제한시간의 reduce-only limit 후 market fallback을 둘 수 있음

### 6.3 청산 우선순위

청산 우선순위는 다음과 같다.

1. `증거금 기준 -5% 하드 스탑`
2. `로컬 ATR 트레일링 청산`
3. `시간 초과 청산`
4. `AI가 제안하는 조기 청산 또는 스탑 강화`

### 6.4 추가로 넣을 만한 종료 규칙

추가 추천:

- `본절 이동`: 가격이 `+1 ATR` 이상 유리하게 가면 손절을 진입가 또는 수수료 반영 소폭 플러스로 이동
- `시간 스탑`: `N`개 캔들 안에 기대 방향으로 못 가면 정리 또는 축소
- `세션 스탑`: 시스템 에러 상태, 스트림 불안정 상태에서는 포지션 축소 또는 강제 정리

버전 1 기본값:

- 본절 이동: 사용
- 시간 스탑: 사용
- 고정 익절가: 사용 안 함

### 6.5 분할매수와 분할매도 기준

실거래 기본값 권장:

- 첫 실거래 버전에서는 `분할매수 비활성화` 권장
- 백테스트로 유의미한 개선이 확인된 뒤에만 활성화
- 진입 시점 고정 초기 증거금 손절을 쓰는 동안 승인된 라이브 기본 버전에서는 `분할매수 비활성화`

추후 분할매수를 켠다면 규칙은 다음과 같다.

- 최대 진입 분할 수: `2` 또는 `3`
- 사전에 정의한 진입 구간 밖에서 손실 중 물타기 금지
- 원래 무효화 조건이 깨진 뒤 추가 진입 금지
- 모든 분할 진입 후에도 총 리스크는 계좌 리스크 한도 안에 있어야 함
- 마틴게일과 무제한 물타기 금지

추가 회계 규칙:

- 각 트랜치는 독립적인 `entry_initial_margin_fixed`, 체결가, 손절 계약을 따로 기록
- 트랜치별 손절 회계와 대사 로직이 완성되기 전에는 라이브 분할매수 활성화 금지

권장 템플릿:

- 1차 진입: 계획 수량의 `50%`를 기본 시그널에서 진입
- 2차 진입: `30%`는 셋업이 유지되고 리클레임, 지지 확인, 거래량 확인 중 하나가 있을 때만 추가
- 3차 진입: `20%`는 가격이 최소 `0.5 ATR` 유리하게 움직이거나 추세 지속이 확인될 때만 추가

아래 상황에서는 남은 분할을 채우지 않는다.

- 스프레드나 슬리피지 확대
- ATR이 허용 구간 이탈
- 손절 거리가 비정상적으로 커짐
- AI 시장국면 게이트가 부정적으로 전환

분할매도는 ATR 트레일과 함께 운영할 수 있다.

- 초기 축소: `+1 ATR` 도달 후 `20% ~ 30%` 축소, 스탑은 본절 부근으로 이동
- 약화 축소: 추세 품질 저하 또는 AI의 방어적 청산 의견 시 `20% ~ 30%` 축소
- 최종 잔량: ATR 트레일링 스탑으로 종료

## 7. 버전 1 전략 골격

전략은 하나만 사용한다.

`추세 추종형 눌림목 진입 + 시장국면/변동성 필터`

### 7.1 롱 진입 조건

아래를 모두 만족해야 한다.

- `15m` 빠른 EMA가 느린 EMA 위
- `1h` 추세 필터가 약세가 아님
- `3m` 눌림이 EMA 구간 또는 VWAP 근처까지 들어옴
- `3m` RSI가 과열에서 중립 부근으로 식었다가 다시 상승
- 현재 캔들이 단기 트리거 위로 다시 마감
- ATR이 허용 구간 안
- AI 시장국면 게이트가 진입을 막지 않음

### 7.2 숏 진입 조건

아래를 모두 만족해야 한다.

- `15m` 빠른 EMA가 느린 EMA 아래
- `1h` 추세 필터가 강세가 아님
- `3m` 되돌림이 EMA 구간 또는 VWAP 근처까지 들어옴
- `3m` RSI가 위로 리셋된 뒤 다시 하락
- 현재 캔들이 단기 트리거 아래로 다시 마감
- ATR이 허용 구간 안
- AI 시장국면 게이트가 진입을 막지 않음

### 7.3 진입 필터

다음 상황에서는 진입하지 않는다.

- ATR이 너무 낮아 시장이 죽어 있음
- ATR이 너무 높아 구조가 불안정함
- 미결제약정 급증이 가격 확인 없이 발생
- 펀딩 흐름이 비정상적임
- 최근 연속 손실로 쿨다운 발동
- 스프레드나 슬리피지가 기준 초과

### 7.4 추가로 넣으면 좋은 필터

- 주요 경제지표 발표 전후 블랙아웃
- 손절 직후 `N`개 캔들 재진입 금지
- 일일 최대 거래 횟수 제한
- 웹소켓 상태가 불안정하면 신규 진입 금지

## 8. Gemma4 학습 및 추론 설계

### 8.1 "비트코인 데이터로 Gemma4 학습"의 현실적 의미

이 프로젝트에서 말하는 학습은:

- 차트 데이터로 기초모델을 처음부터 새로 만드는 것 아님
- BTC 시장 데이터셋을 구축하고
- `LoRA` 또는 `QLoRA` 형태로 미세조정하거나 적응시키고
- 시장 국면과 셋업 품질을 판단하도록 만드는 것

### 8.2 사용할 로컬 모델

로컬 추론에 사용할 기본 대상은 다음과 같다.

- 모델: `Jiunsong/supergemma4-26b-uncensored-mlx-4bit-v2`
- 모델 카드에 표시된 base model: `google/gemma-4-26B-A4B-it`
- 형식: MLX 4-bit
- 운영 방식: Apple Silicon에서 로컬 텍스트 추론 서버

이 모델은 다음 용도로 본다.

- 시장국면, 셋업 품질, 청산 자세 판단용 로컬 추론 엔진
- 데이터와 리뷰 파이프라인이 안정화된 뒤의 오프라인 미세조정 대상

### 8.3 학습 목표

추천 학습 대상:

- `market_regime`: 상승추세, 하락추세, 횡보, 노이즈, 돌파위험
- `setup_quality`: 강함, 보통, 약함
- `entry_veto`: 진입 허용, 거부, 사이즈 축소
- `exit_posture`: 홀드, 스탑 강화, 청산

버전 1에서는 AI가 직접 주문 명령을 생성하게 하지 않는다.

### 8.4 학습 샘플 설계

샘플 1개는 구조화된 시장 윈도우여야 한다.
예를 들면:

- 최근 `3m`, `15m`, `1h` 요약 캔들 정보
- ATR, EMA 기울기, RSI, ADX, VWAP 거리
- 펀딩비와 변화량
- 미결제약정 변화
- 테이커 매수/매도 비율
- 현재 추세 상태
- 필요하면 포지션 상태

라벨 출처:

- 룰 엔진이 만든 태그
- 백테스트 결과
- 수동 리뷰한 고품질 거래 샘플

모든 학습 샘플에는 아래 버전 키도 같이 저장한다.

- `policy_version`
- `strategy_version`
- `feature_schema_version`
- `labeler_version`
- `dataset_version`
- `model_base`
- `adapter_version`

### 8.5 데이터 분할 원칙

데이터 누수를 막기 위해:

- 학습/검증/테스트를 시간 기준으로 분할
- 미래 구간 데이터를 과거 학습에 섞지 않음
- 보지 않은 날짜 구간으로 평가
- 총수익만 보지 말고 시장국면별 성능 확인

### 8.6 추론 입력 형식

실시간 추론에는 원시 차트가 아니라 구조화된 JSON을 넣는다.

예시 입력 묶음:

- 최근 OHLCV 요약
- 타임프레임별 지표 스냅샷
- ATR, EMA 기울기, RSI, ADX, VWAP 거리
- 펀딩비
- 미결제약정 변화
- 테이커 매수/매도 비율
- 최근 스윙 구조
- 현재 포지션과 미실현 손익
- 진입 후 경과 시간

### 8.7 출력 형식

실거래 경로에서는 자유문장 출력 대신 엄격한 JSON만 허용한다.

```json
{
  "regime": "trend_up|trend_down|range|high_volatility|unclear",
  "setup_quality": 0.0,
  "entry_action": "allow|veto|reduce_size",
  "exit_action": "hold|tighten_stop|full_exit",
  "confidence": 0.0,
  "reason_codes": ["trend_alignment", "weak_volume"]
}
```

### 8.8 로컬 추론 실행 시점

AI는 의미 있는 이벤트에서만 실행한다.

- 실행 타임프레임 캔들 마감
- 신규 셋업 후보 발생
- 포지션 상태 변화
- ATR 스탑 업데이트 판단 시점

매 틱마다 돌리면 안 된다.

### 8.9 AI 안전 규칙

AI 출력이 이상하면:

- 무시
- 룰 기반 로직으로 복귀
- 실패 로그 기록

신뢰도가 낮으면:

- 추천 무시

AI가 리스크를 늘리자고 하면:

- 거부

## 10. 백테스트 및 검증

### 10.1 백테스터 필수 기능

- 시그널용 계약가격 캔들 + 손절용 저주기 마크프라이스 재생을 함께 쓰는 하이브리드 리플레이
- 수수료 모델
- 슬리피지 모델
- 레버리지 반영 사이징
- 진입 시점 고정 초기 증거금 대비 `-5%` 손절
- 마크프라이스 기준 미실현 손익 트리거
- ATR 트레일링 청산
- 시간 스탑
- 쿨다운 규칙
- 일일 손실 제한

권장 리플레이 방식:

- 시그널 엔진은 원래 실행 타임프레임 예: `3m`에서 동작
- 마크프라이스 손절 탐지는 더 낮은 주기 예: `1m` 또는 더 세밀한 데이터에서 수행
- 세밀한 데이터가 없으면 close-only 대신 보수적인 intrabar breach 규칙 사용

보수적 fallback 규칙:

- 저주기 마크프라이스 고가/저가가 손절 임계값을 intrabar로 넘기면 손절 발동으로 간주
- 체결가는 손절가와 다음 실행 가능한 계약가격 중 더 불리한 값에 슬리피지까지 반영

### 11.2 주문 생명주기

권장 순서:

1. 유효한 셋업 탐지
2. 허용 노셔널과 수량 계산
3. 증거금 기반 하드 스탑 가격 계산

5. 체결 확인
6. 즉시 보호 스탑 등록
7. 포지션 보유 중 ATR 트레일 평가 시작

### 11.3 스탑 업데이트 규칙

- 포지션이 열려 있는 동안 보호 스탑 제거 금지

- 기본적으로 ATR 청산 로직이 하드 손절 계약 자체를 덮어쓰면 안 됨
- 추후 안전 강화 로직을 추가하더라도 하드 손절은 더 타이트하게만 조정 가능
- 스탑 업데이트 실패 시 신규 진입 중단 및 경고

### 11.4 거래소 상태 대사

주기적으로 아래를 맞춘다.

- 로컬 포지션 vs 거래소 포지션
- 로컬 미체결 주문 vs 거래소 미체결 주문
- 기대 손절가 vs 실제 보호 스탑
- 레버리지/마진모드 설정값

불일치 시:

- 신규 진입 중단
- 불일치 해소 후 재개

최소 표시 항목:

- 최근 거래
- 현재 포지션
- 일일 손익
- 현재 활성 스탑
- 전략 상태
- 리스크 상태
- 마지막 AI 판단

### 13.4 손실 원인 자동분석 파이프라인

모든 손실 거래는 자동으로 리뷰해야 한다.

각 손실 거래마다 아래를 계산해 저장한다.

- 진입 시점과 청산 시점의 시장 국면
- 룰 조건이 실제로 모두 충족되었는지 여부
- AI가 동의했는지, 거부했는지, 무시됐는지 여부
- 슬리피지와 수수료 기여도
- 최대 유리 구간과 최대 불리 구간
- 실제 변동성 대비 하드 스탑이 너무 타이트했는지 여부
- 진입이 빨랐는지, 늦었는지, 구조적으로 잘못됐는지 여부

원인 코드는 먼저 규칙적으로 붙인다. 예시:

- `regime_misread`
- `volatility_expansion`
- `false_breakout`
- `entered_too_late`
- `overtrading`
- `slippage_too_high`
- `stop_too_tight`
- `execution_failure`
- `ignored_cooldown`
- `model_veto_missed`

AI는 2차 설명 레이어를 추가할 수 있지만, 1차 진실원은 항상 규칙 기반 태그여야 한다.

### 13.5 자동 백테스트와 재학습 루프

권장 개선 루프는 다음과 같다.

1. 완료된 거래, 피처 스냅샷, AI 출력 저장
2. 손실 거래를 규칙 기반으로 자동 태깅
3. AI가 손실 패턴을 요약 설명
4. 일간 또는 주간 오프라인 배치로 반복 손실 패턴 군집화
5. 후보 룰 변경이나 파라미터 변경안 생성
6. 해당 후보를 백테스트 실행
7. 아웃오브샘플과 워크포워드로 다시 검증
8. shadow 또는 paper 모드에서 최소 관찰 기간 검증
9. 통과한 경우에만 학습 데이터셋에 반영
10. 모델은 오프라인에서만 재학습 또는 갱신
11. 실거래 봇이 자기 룰을 실시간으로 직접 바꾸는 것은 금지
12. 새로 학습된 모델을 실제 자금에 즉시 자동 반영하는 것은 금지

변경 승인 기준:

- 수수료와 슬리피지 반영 후 개선
- 최대 낙폭이 허용 한도 이상 악화되지 않을 것
- 여러 날짜 구간과 시장 국면에서 안정적일 것
- 꼬리손실이 증가하지 않을 것

권장 반영 지연 규칙:

- 새 모델 또는 새 룰 패키지는 최소 주간 단위보다 더 빠르게 승격하지 않음
- 실거래 승격 전 최소 `7일` shadow 또는 paper 관찰 기간을 둠
- 재학습이나 승격 전에 최소 종료 거래 표본 수 기준을 둠

### 13.6 룰 보존과 승격 거버넌스

리뷰가 과도해져서 원래 규칙을 잠식하지 않도록 시스템을 3개 층으로 분리한다.

1. `Immutable baseline policy`
- 하드 안전 규칙과 기본 거래 제약
- 리뷰 루프가 수정할 수 없음

2. `Experiment candidates`
- 파라미터 변경안, 필터 변경안, 모델 변형안
- 오프라인 백테스트, 섀도우 런, 페이퍼 모드에서만 검증

3. `Approved live versions`
- 명시적 버전 번호가 붙은 수동 승인 전략 패키지
- 실제 자금으로 실행 가능한 유일한 버전

필수 거버넌스 규칙:

- 모든 라이브 전략은 버전 id와 변경 이력을 가져야 함
- 코어 룰은 실험 결과와 분리 저장해야 함
- 후보 변경안이 마지막 승인 버전을 덮어쓰면 안 됨
- 모든 거래, 학습 샘플, 모델 산출물은 명시적 버전 메타데이터를 가져야 함
- 룰 기반 단독 거래도 필드를 비우지 말고 sentinel 값으로 메타데이터를 남겨야 함
- 한 번의 리뷰 사이클에서 큰 전략 변경은 1개만 승격
- 모든 승격은 고정된 과거 구간 회귀 테스트를 통과해야 함
- 모든 승격은 직전 승인 버전으로 롤백할 수 있어야 함
- 새 버전이 코어 룰을 위반하면 자동 거절

운영 원칙:

- 리뷰는 조언자
- 오프라인 연구는 실험장
- 라이브 정책 변경은 명시적 승인 대상

## 14. 권장 모듈 구조

- `config/`

- `ai/`
- `risk/`

역할 분리:

- `policy/`: 불변 라이브 룰, 승인된 전략 버전, 롤백 대상 관리
- `data/`: REST/WebSocket 수집과 저장
- `features/`: 지표 및 피처 스냅샷 생성
- `strategy/`: 기계적 셋업 탐지
- `ai/`: Gemma4 학습데이터 준비, 추론, JSON 파싱
- `risk/`: 사이징, 손절 규칙, 킬 스위치
- `execution/`: 주문 실행과 상태 대사
- `backtest/`: 시뮬레이션과 리포트
- `runtime/`: 스케줄링과 오케스트레이션
- `monitoring/`: 로그, 알림, 요약
- `review/`: 손실 분석, 원인 태깅, 재학습 데이터셋 생성

권장 policy 저장 분리:

- `policy/approved_live/`
- `policy/experiment/`
- `policy/training_candidates/`
- `policy/rollback/`

## 16. 버전 1에서 고정할 결정

- 심볼: `BTCUSDT`
- 실행 타임프레임: `3m`
- 확인 타임프레임: `15m`
- 상위 필터: `1h`
- 모드: one-way
- 마진: isolated
- 실거래 시작 레버리지: `2x`
- 시스템 레버리지 상한: `10x`
- 동시 포지션: 1개
- 진입 스타일: 추세 추종 눌림목
- 하드 손절 기준: 진입 시점 고정 초기 증거금의 `-5%`
- 하드 손절 트리거: `mark price` 기준 미실현 손익
- 거래소 손절 트리거 모드: `workingType=MARK_PRICE`
- 비상 청산: 마크프라이스 위반 확인 시 reduce-only 시장가 정리
- 시그널과 ATR 소스: 계약가격 캔들
- 수익 보호: 계약가격 캔들 기준 ATR 트레일링 스탑
- 분할매수: 기본은 비활성화, 백테스트 근거 있을 때만 활성화
- 분할매도: `+1 ATR` 이후 `20% ~ 30%` 축소 옵션 허용
- AI 역할: 시장국면 게이트와 셋업 품질 필터
- 라이브 모델 반영: 지연 승격, 즉시 반영 금지

## 18. 최종 구조

권장 아키텍처는 다음과 같다.

`Data -> Features -> Rule Strategy -> AI Gate -> Policy -> Risk Manager -> Execution -> Monitoring -> Review`

좋은 버전 1의 기준은 "가장 똑똑한 봇"이 아니다.
리뷰가 개선을 도울 수는 있어도, 안전 계약 자체를 지우면 안 된다.
기준은 아래다.

- 항상 손절 보호가 존재함
- 규칙대로 정확히 청산함
- API와 웹소켓 장애를 버팀
- 로그가 재현 가능함
- 백테스트를 정직하게 할 수 있음
- 이후 AI를 안전하게 더 얹을 수 있음
